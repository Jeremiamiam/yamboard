# Roadmap: GET BRANDON — Web App

## Overview

Cinq phases livrent l'interface web qui remplace le CLI GBD. La fondation DB et l'auth arrivent en premier car tout le reste en dépend. Le streaming AI — risque technique le plus élevé du projet — est prouvé en Phase 2 avant de construire l'UI autour. La Phase 3 ajoute l'UI workflow et le dashboard. La Phase 4 ferme la boucle avec l'upload de fichiers et l'extraction des outputs JSON. La Phase 5 finalise avec le wiki HTML et le polish.

## Phase Details

### Phase 1: Foundation + Auth
**Goal**: Les deux utilisateurs peuvent se connecter et voir/créer leurs projets
**Depends on**: Nothing (first phase)
**Requirements**: AUTH-01, AUTH-02, AUTH-03, AUTH-04, PROJ-01, PROJ-02, PROJ-03, PROJ-04
**Success Criteria** (what must be TRUE):
  1. Utilisateur peut se connecter avec email/mot de passe et rester connecté après rechargement de page
  2. Utilisateur non connecté est automatiquement redirigé vers /login
  3. Utilisateur peut créer un nouveau projet client avec un nom et le voir apparaître dans le tableau de bord
  4. Utilisateur peut rouvrir un projet existant depuis le tableau de bord
  5. Le tableau de bord affiche quels workflows ont été complétés pour chaque projet
**Plans**: 5 plans

Plans:
- [x] 01-01-PLAN.md — Project scaffold: Next.js 16 app bootstrap + Supabase install + env config
- [x] 01-02-PLAN.md — Auth infrastructure: Supabase clients (browser/server) + proxy.ts session middleware
- [x] 01-03-PLAN.md — Login page: email/password form + signInWithPassword + signOut action
- [x] 01-04-PLAN.md — Dashboard + project CRUD: DB schema, project list, create/delete modal, Server Actions
- [x] 01-05-PLAN.md — Project detail + workflow status UI + Phase 1 human verification

### Phase 2: Core Streaming
**Goal**: Un workflow GBD peut être lancé et sa réponse s'affiche token par token sans timeout
**Depends on**: Phase 1
**Requirements**: CHAT-01, CHAT-02, CHAT-03, CHAT-04, CHAT-05, CHAT-06
**Success Criteria** (what must be TRUE):
  1. L'interface affiche les messages utilisateur et agent en alternance dans un fil de conversation
  2. Les tokens de la réponse agent apparaissent en temps réel (streaming visible, pas de bloc d'un coup)
  3. Un workflow de 3-5 minutes se termine sans timeout ni erreur de connexion
  4. Un indicateur visuel s'affiche pendant la latence initiale (avant le premier token)
  5. Si le stream SSE s'interrompt, la reconnexion se fait automatiquement et l'historique complet est rechargé à la reprise
**Plans**: 4 plans

Plans:
- [ ] 02-01-PLAN.md — Infrastructure: npm packages + Supabase messages table + DB helpers + Tailwind typography plugin
- [ ] 02-02-PLAN.md — API Route Handler: POST /api/chat SSE streaming (Anthropic SDK + maxDuration=300)
- [ ] 02-03-PLAN.md — Client hook: useChatStream avec SSE reader + reconnexion backoff exponentiel
- [ ] 02-04-PLAN.md — UI: MessageBubble + TypingIndicator + ReconnectionBanner + ChatInterface + page chat + vérification humaine

### Phase 3: UI Workflow + Dashboard
**Goal**: L'utilisateur peut sélectionner et lancer n'importe quel workflow depuis l'interface, avec feedback visuel sur les dépendances
**Depends on**: Phase 2
**Requirements**: WF-01, WF-02, WF-03, WF-04, WF-05, WF-06
**Success Criteria** (what must be TRUE):
  1. Utilisateur peut lancer le workflow /start depuis un projet
  2. Les workflows /platform, /campaign, /wireframe sont visuellement désactivés tant que leurs prérequis ne sont pas complétés
  3. Utilisateur peut lancer /site-standalone indépendamment des autres workflows
  4. Une fois /start complété, /platform devient disponible et ainsi de suite pour la chaîne de dépendances
**Plans**: 4 plans

Plans:
- [x] 03-01-PLAN.md — Workflow system prompts + API route extension: src/lib/workflows/system-prompts.ts + workflowSlug routing in /api/chat
- [x] 03-02-PLAN.md — WorkflowLauncher component + project page: interactive launch buttons per workflow status
- [x] 03-03-PLAN.md — Chat-side workflow wiring: markWorkflowComplete action + useChatStream/ChatInterface/chat page extensions
- [x] 03-04-PLAN.md — Build check + human verification of complete Phase 3 workflow UI

### Phase 4: File Upload + Output Extraction
**Goal**: Les fichiers clients peuvent être uploadés et les outputs JSON des workflows sont automatiquement sauvegardés et téléchargeables
**Depends on**: Phase 3
**Requirements**: FILE-01, FILE-02, FILE-03, FILE-04, OUT-01, OUT-04
**Success Criteria** (what must be TRUE):
  1. Utilisateur peut uploader un PDF, .txt, .md ou .docx dans un projet via l'interface (drag/drop ou click)
  2. Les fichiers uploadés sont disponibles pour l'agent lors du workflow /start
  3. Les fichiers de type non autorisé sont rejetés avec un message d'erreur clair
  4. Les JSON produits par les workflows (CONTRE-BRIEF.json, PLATFORM.json, etc.) sont sauvegardés automatiquement et téléchargeables

**Goal**: Le wiki HTML généré est consultable directement dans l'interface et les outputs sont accessibles sans friction
