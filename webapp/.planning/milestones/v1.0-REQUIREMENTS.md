# Requirements: GET BRANDON Web App

**Defined:** 2026-02-25
**Core Value:** N'importe quel membre de l'équipe peut démarrer et piloter un projet GBD sans passer par le CLI.

## v1 Requirements

### Authentication

- [x] **AUTH-01**: Utilisateur peut se connecter avec email/mot de passe (Supabase Auth)
- [x] **AUTH-02**: Session persistée entre les rechargements de page (JWT refresh via middleware)
- [x] **AUTH-03**: Utilisateur non connecté est redirigé vers /login
- [x] **AUTH-04**: Pas d'inscription publique — 2 comptes créés manuellement dans Supabase

### Projets

- [x] **PROJ-01**: Utilisateur peut créer un nouveau projet client avec un nom
- [x] **PROJ-02**: Utilisateur voit la liste de tous les projets (tableau de bord)
- [x] **PROJ-03**: Utilisateur peut rouvrir un projet existant et reprendre la conversation
- [x] **PROJ-04**: Chaque projet affiche quels workflows ont été complétés (start ✓, platform ✓, etc.)

### Chat & Streaming

- [x] **CHAT-01**: Interface de conversation avec messages alternés user/agent
- [x] **CHAT-02**: Les réponses de l'agent s'affichent en streaming (token par token)
- [x] **CHAT-03**: Le stream survit aux workflows longs (3-5 minutes) sans timeout
- [x] **CHAT-04**: Indicateur visuel pendant la latence initiale (avant le premier token)
- [x] **CHAT-05**: Reconnexion automatique si le stream SSE s'interrompt
- [x] **CHAT-06**: Historique complet de la conversation persisté en DB et rechargé à la reprise

### Workflows

- [x] **WF-01**: Utilisateur peut lancer le workflow `/start` depuis un projet
- [x] **WF-02**: Utilisateur peut lancer le workflow `/platform` si `/start` est complété
- [x] **WF-03**: Utilisateur peut lancer le workflow `/campaign` si `/platform` est complété
- [x] **WF-04**: Utilisateur peut lancer le workflow `/site-standalone` (indépendant)
- [x] **WF-05**: Utilisateur peut lancer le workflow `/wireframe` si `/site-standalone` est complété
- [x] **WF-06**: Les workflows non disponibles sont visuellement désactivés (dépendances respectées)

### Upload de fichiers

- [ ] **FILE-01**: Utilisateur peut uploader des fichiers (PDF, .txt, .md, .docx) dans un projet
- [ ] **FILE-02**: Upload direct client → Supabase Storage (bypass serverless function)
- [ ] **FILE-03**: Les fichiers uploadés sont accessibles à l'agent lors du workflow `/start`
- [ ] **FILE-04**: Validation du type de fichier côté client et serveur

### Outputs & Livrables

- [ ] **OUT-01**: Les JSON produits (CONTRE-BRIEF.json, PLATFORM.json, etc.) sont sauvegardés en DB
- [ ] **OUT-02**: Le wiki HTML (GBD-WIKI.html) est stocké et affiché dans l'interface via iframe
- [ ] **OUT-03**: Le wireframe HTML (STANDALONE-WIREFRAME.html) est stocké et affiché via iframe
- [ ] **OUT-04**: Utilisateur peut télécharger les fichiers JSON et HTML produits

## v2 Requirements

### Partage client

- **SHARE-01**: Lien de partage unique par projet (lecture seule) pour le client
- **SHARE-02**: Le client peut consulter le wiki sans compte

### Contrôle des coûts

- **COST-01**: Dashboard d'usage tokens par projet et par workflow
- **COST-02**: Prompt caching Anthropic pour les system prompts répétés

### Export

- **EXP-01**: Export PDF du wiki
- **EXP-02**: Export formaté pour Figma Slides (PITCH, PLATFORM)

## Out of Scope

| Feature | Raison |
|---------|--------|
| Inscription publique / SaaS | Outil interne uniquement — 2 users hardcodés en v1 |
| Application mobile | Web-first |
| Workflows `/social`, `/retro`, `/brandbook` | Pas encore implémentés en CLI |
| Notifications email | Pas de cas d'usage identifié en v1 |
| Collaboration temps réel (curseurs) | Sur-engineering pour 2 utilisateurs |
