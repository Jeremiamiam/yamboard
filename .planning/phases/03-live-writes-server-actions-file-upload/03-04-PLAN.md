**Gestion is_primary (Pitfall 7 de RESEARCH.md):**

Pour `createContact` avec `isPrimary=true`:
```typescript
// Reset AVANT l'insert
await supabase
  .from('contacts')
  .update({ is_primary: false })
  .eq('client_id', params.clientId)
  .eq('owner_id', user.id)
// Puis INSERT avec is_primary: true
```

Pour `updateContact` avec `isPrimary=true`:
```typescript
// Fetch client_id du contact à mettre à jour
const { data: existing } = await supabase
  .from('contacts')
  .select('client_id')
  .eq('id', contactId)
  .eq('owner_id', user.id)
  .single()

// Reset is_primary sur tous les autres contacts du client
await supabase
  .from('contacts')
  .update({ is_primary: false })
  .eq('client_id', existing.client_id)
  .eq('owner_id', user.id)

// Puis UPDATE le contact cible
```

Pour `deleteContact`: fetch `client_id` avant la suppression pour construire `revalidatePath('/[clientId]', 'page')` avec le bon ID.

Règles identiques aux autres actions: `getUser()`, `owner_id: user.id`, `.eq('owner_id', user.id)` sur UPDATE/DELETE, `revalidatePath()` après mutation.
